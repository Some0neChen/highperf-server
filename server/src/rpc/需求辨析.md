太有必要了。在写下第一行代码之前，把 API 契约（Contract）和架构扩展点白纸黑字地定死，能帮你省去后期无数的返工。

基于我们确定的“窄版 Radix Tree 路由 + 裸 Body + 后端 RPC”的最优解，下面是这套架构下最终定型的报文格式和未来的扩展指南。

---

### 一、 最终敲定的 CRUD 报文格式 (面向客户端/前端)

你的 C++ 网关对外暴露的接口将极致简洁，**不解析任何 Query (`?k=v`)，也不依赖 JSON 库**。

#### 1. Create & Update (新增与更新 - 统一使用 PUT)

*语义：幂等操作，Key 不存在则创建，存在则覆盖更新。*

* **HTTP 请求 (客户端 -> 网关):**
```http
PUT /kv/TableName/Elizabeth HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: text/plain
Content-Length: 9

beautiful

```


* **网关内部解析结果:**
* `path_params["table"]` = `TableName`
* `path_params["key"]` = `Elizabeth`
* `req->body` = `beautiful`


* **转换为下游 RPC 请求:** 组装成 TLV 报文发送给 KV Server，调用 `KV.Put` (或 `KV.Set`) 方法。

#### 2. Read (读取 - 使用 GET)

* **HTTP 请求 (客户端 -> 网关):**
```http
GET /kv/TableName/Elizabeth HTTP/1.1
Host: 127.0.0.1:8080

```


* **网关内部解析结果:**
* `path_params["table"]` = `TableName`
* `path_params["key"]` = `Elizabeth`
* `req->body` = `(空)`


* **转换为下游 RPC 请求:** 组装成 TLV 报文发送给 KV Server，调用 `KV.Get` 方法。
* **HTTP 回包示例 (网关 -> 客户端):**
```http
HTTP/1.1 200 OK
Content-Type: text/plain
Content-Length: 9

beautiful

```



#### 3. Delete (删除 - 使用 DELETE)

* **HTTP 请求 (客户端 -> 网关):**
```http
DELETE /kv/TableName/Elizabeth HTTP/1.1
Host: 127.0.0.1:8080

```


* **网关内部解析结果:**
* `path_params["table"]` = `TableName`
* `path_params["key"]` = `Elizabeth`


* **转换为下游 RPC 请求:** 组装成 TLV 报文，调用 `KV.Delete` 方法。

### 二、 架构扩展点：后续如何接入新业务？

这套架构的精髓在于“控制复杂度”。当未来需要扩展时，代码的变动点是非常明确且集中的。

#### 扩展场景 1：KV 服务新增一个特殊的管理接口

**需求：** 需要一个接口来获取当前 KV 引擎的状态（如内存占用、Key 的总数）。
**扩展点：HTTP Router 路由表 + 新增 Handler**

1. 写一个 `HttpKvStatHandler`，里面封装发送状态查询 RPC 报文的逻辑。
2. 在 WebServer 启动初始化时，增加一行精确路由注册：
```cpp
// 注意：这里的静态路由 /kv/stat 会在 Radix Tree 中优先于 /kv/:table 命中
router.add("GET", "/kv/stat", std::make_unique<HttpKvStatHandler>());

```



**代码变动范围：** 极小。不影响任何现有的增删改查逻辑。

#### 扩展场景 2：完全引入一个新的微服务（例如：配置中心服务 ConfigServer）

**需求：** 开发了一个全新的后端 C++ 进程，跑在 9005 端口，负责下发系统配置。
**扩展点：泛型 RPC 入口 + 服务发现表**
对于这种非资源型（非 RESTful）的复杂微服务，直接走我们的泛型 RPC 通道。

1. **前端调用约定：** 统一使用 `POST /rpc/config/pull`。
2. **网关层的改动：** HTTP Router 层**一行代码都不需要改**！因为 `/rpc/:service/:method` 这条规则天生就能接住它。
3. **唯一需要扩展的地方：** 在网关层的 `ServiceRegistry`（服务注册表/配置字典）中，新增一行映射，告诉网关 `config` 服务在哪个端口：
```cpp
// 伪代码：网关在转发前查表
service_map["kv"] = "127.0.0.1:9001";
service_map["config"] = "127.0.0.1:9005"; // 新增这一行即可

```



**代码变动范围：** 几乎为零（仅需改配置）。网关提取出 `service="config"` 后，自动去连接 `9005` 端口并完成 TLV 转发。

#### 扩展场景 3：KV 服务需要支持批量插入（BatchPut）

**需求：** 前端需要一次性写入 100 个 Key-Value 字典。这种场景用上面的 URL Path Param 已经无法满足了，必须传复杂的 JSON。
**扩展点：回退到泛型 RPC**
不要去改你优雅的 `PUT /kv/:table/:key` 接口，也不要去写复杂的 HTTP Body Parser。直接让前端走 RPC 入口：

* **请求：** `POST /rpc/kv/batch_put`
* **Body：** `[{"k":"1","v":"a"}, {"k":"2","v":"b"}]`
* **网关：** 提取出 `kv` 和 `batch_put`，把那坨巨大的 JSON 数组当成不透明的字符串 Payload 丢给后端的 C++ KV Server，让 KV Server 自己用 JSON 库去慢慢反序列化。
